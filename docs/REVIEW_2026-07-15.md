*Review date: 2026-07-15. Reviewed at commit `519f687` (stream discovery / window-boundary
parser fix). Scope: whole repo — `bridge/`, `firmware/`, `tests/`, `docs/` — read against the
stated goal: an open-source consumer product for pleasant, inspiring personal use; Palm-like,
charming, deliberately lighter than the mainstream phone ecosystem.*

## 1. Executive summary

This project is in unusually good shape for its stage. The two hardest problems — a lossless
Palm↔DAV codec and a two-way sync engine that fits beside TLS in ~80 KB of heap — are solved,
tested, and *provable offline* (12 host gates, several of which deliberately reproduce
device-only failure modes like fetch-buffer truncation and window-boundary parsing on the
host). The engineering log discipline (`BUILD_PROGRESS.md`, `NEXT_STEPS.md`) is a real asset.

The biggest gaps are no longer in the sync engine. They are:

1. **Product/legal hygiene** — there is **no `LICENSE` file** despite the firmware's GPLv3
   obligation (PumpkinOS fonts/icons), no CI, no user-facing setup docs. These block "open
   source consumer product" more than any bug does, and all are fixable in-repo.
2. **First-run intuitiveness** — a new user must hand-edit `config.ini` or enter a 19-character
   app-specific password via an *untuned* Graffiti recognizer with no ink feedback. That is
   the single biggest gap between "our prototype" and "a device someone else can love."
3. **Charm details** — the bones are authentic (real Palm fonts/icons, mono theme, silkscreen,
   categories, Day/Month views), but the moments Palm users remember — visible Graffiti ink,
   the HotSync dialog, the little sounds, the easter eggs — aren't there yet. These are cheap
   and most of them are buildable off-device (see §6).
4. **Memory management** is fundamentally sound (streaming everywhere, mode-multiplexing,
   disk-backed reconcile) but has drifted from its own stated rule: **~21 KB of sync-only
   buffers are back in permanent BSS**, and the "tear down the UI during sync" mode-switch
   from the roadmap was never implemented — sync currently runs ~3 KB from the measured peak.

## 2. Memory management

### 2.1 What's working (keep doing this)

- **Streaming as a design rule.** PDB read/write is one-record-at-a-time; the reconcile is a
  disk-backed 3-way merge-join; DAV enumeration responses spool to SD and are parsed in a
  sliding window (`dav_parse_*_stream`). Peak RAM is O(1) in record count. This is the right
  architecture for this chip, and `bigsync.c` / `streamparse.c` / `idempotent.c` prove it on
  the host with device sizing — an excellent pattern.
- **Time-multiplexing the two heavy consumers** (LVGL vs Wi-Fi+TLS) instead of summing them,
  with `dav_disconnect()` deliberately called before every heap-heavy sort so the ~40 KB TLS
  working set never coexists with a sort. The keep-alive change (part 9) kept this invariant.
- **The failure-mode notes in comments** — the `lv_bar` draw-layer WDT, the `lv_list` object-
  pool exhaustion, the 32 KB stack rationale in `hotsync_start()` — are genuinely valuable.
  Each one is a landmine map for future contributors. Keep writing them.
- **Defensive degradation**: `build_record_table()` shows "(low memory)" instead of crashing;
  `sync_collection` returns -1/-2 instead of taking the UI down; the empty-result overwrite
  guard (rc -2) protects user data.

### 2.2 Findings and recommendations

**M1 — ~21 KB of sync-only buffers are permanently resident (BSS), violating the U0 rule.**
`bridge/sync.c` holds `g_body[8192]`, `g_lrec[4096]`, `g_objbuf[8192]` and two `static
uint8_t ai[512]` — all sync-lifetime data, all resident even in interactive mode. The
UI_ROADMAP explicitly called converting sync arenas static→heap "a prerequisite, not
optional" (U0), and the comments justify BSS only as "keeps the task stack small," which a
sync-start `malloc`/sync-end `free` achieves equally well. **Recommendation:** bundle them
into one `SyncScratch` struct allocated in `sync_collection()` entry and freed on exit
(host-gated; zero device risk to *write*, needs one flash to *verify*). That hands ~21 KB
back to interactive mode — more than the entire LVGL draw buffer.

**M2 — The roadmap's mode-switch (tear down LVGL buffers during sync) was never built, and
you're running ~3 KB from the edge.** README: "WiFi+TLS+sync while LVGL is up ≈ 166 KB free
vs ~169 KB peak." Today the 19 KB draw buffer, the 24 KB LVGL pool, and all widget trees stay
live during HotSync. It works on your bench with your data; a user with a photo-heavy vCard
or a long cert chain is the person who finds the edge. **Recommendation:** implement the
sync-mode screen from UI_ROADMAP §"The transition": on `hotsync_start`, optionally free the
draw buffer (or at minimum the current screen's widget tree, keeping only the status label),
rebuild on completion. Combined with M1 that's ~40 KB of new headroom during the exact window
that has almost none. (Design + code can be written and compile-checked off-device; the
verify needs hardware.)

**M3 — Wi-Fi up/down churn across many syncs is an untested leak/fragmentation surface.**
`wifi_up()`/`wifi_down()` re-init and deinit the whole Wi-Fi driver per sync (correct for the
budget), but ESP-IDF's netif/event-loop objects are not all reclaimed symmetrically
(`esp_netif_create_default_wifi_sta` is intentionally kept in `s_netif`, `esp_netif_init` is
re-called every cycle). **Recommendation:** add a cheap invariant now — log
`esp_get_free_heap_size()` + `largest_free_block` at the *same point* before each sync and
after `wifi_down()`, and treat a downward trend across 20 syncs as a bug. A soak test ("sync
every 5 min overnight") should be a standing hardware gate before any release.

**M4 — `MAX_DISC 12` silently drops collections.** A user with more than 12 calendars +
reminder lists + address books (common for families/work accounts) loses the trailing ones
from the Discover screen with no indication — the same *silent truncation* class you've been
systematically eliminating from the DAV layer. Each `DiscColl` is small; **raise to 24–32, or
show "…and N more (narrow your iCloud account)" when the cap hits.** One-line, host-buildable.

**M5 — Fixed 8 KB `OBJ_FETCH_CAP` is handled but worth surfacing to the user.** The deferral
logic (part 4) correctly prevents data loss when an object exceeds the fetch buffer, but the
record then never syncs, forever, invisibly. **Recommendation:** count deferrals in
`SyncStats` and surface "2 items too large to sync" in the Done message — turning a silent
gap into an honest one. (Engine + status-string change; host-gated.)

**M6 — Watchpoints worth adding, not changing:** the 32 KB heap-allocated task stack is the
single largest contiguous allocation in the system; if fragmentation ever makes it fail,
`hotsync_start` already degrades cleanly ("Could not start sync task") — good. Consider
`uxTaskGetStackHighWaterMark` logging at sync end so the 32 KB number is data, not folklore.

### 2.3 Verdict

No red flags; the architecture is right and the discipline is real. The work is to *finish
the stated design* (M1, M2) and instrument the edges (M3, M6) rather than to redesign
anything.

## 3. UX charm & authenticity

The foundation is authentically Palm where it counts: real bitmap fonts and tAIB icons, flat
mono theme with square corners, silkscreen buttons flanking a two-zone Graffiti area,
top-right category trigger, Day/Month date-centric Date Book, delete confirmations, Look Up
in Address. That's the hard 80%. What follows is the 20% that makes people smile.

**C1 — Graffiti has no ink.** The writing pads render nothing while you write. On a real
Palm, the ink trail *was* the experience — without it, users can't tell whether a stroke
registered, where it went wrong, or even where the pad boundaries are. This is the single
highest-leverage charm *and* usability fix in the project. Options in RAM order: (a) an LVGL
canvas over the strip — 168×106 @ 1bpp indexed ≈ 2.3 KB, well within interactive-mode budget;
(b) direct `display_blit` of line segments in the touch path (zero LVGL involvement, like the
calibration crosshairs). Also show the recognized character for ~500 ms in the strip (Palm
did this on some models; your telemetry already computes everything needed).

**C2 — HotSync deserves its dialog.** The HotSync moment was iconic: the logo, the two
rotating arrows, the per-conduit lines ("Synchronizing Date Book…"), the ta-da. Today it's a
button and a text label. You already know `lv_bar` deadlocks mid-sync — fine: the authentic
Palm progress UI was *text plus the logo*, not a bar. A static HotSync logo (you already have
the icon), per-collection conduit lines driven by the existing status string, and the % you
already compute would read as dramatically more authentic *and* dodge the layer-alloc
problem entirely.

**C3 — Sound.** PalmOS clicked on taps, chirped on HotSync start/finish, and alarmed on
Date Book events. The CYD has a speaker output (GPIO26 on the 2432S028R). A tiny square-wave
"click"/"chime" driver (LEDC or DAC, a few dozen lines) with a Prefs on/off toggle would add
more perceived personality per byte than anything else on the backlog. (Alarm playback also
unlocks Date Book alarms actually *alarming* — the data is already synced via VALARM.)

**C4 — Details/forms don't follow the Palm form contract yet.** Palm's record form is
Done/Details/New across the *bottom*; dialogs are OK-right/Cancel-left. Your edit form puts
Cancel/Category/Save at the *top* and Details currently only picks a category. This is F4/F5
from the roadmap, still open — worth doing before the look calcifies, because it touches
every app. Also missing Palm staples: **Edit Categories…** (rename/add — the AppInfo codec
already supports it), event **alarm/repeat** in Details, and Address showing more than 5
fields (data survives round-trips; the user just can't see or edit it on-device).

**C5 — Ship the About box with a soul, and hide the scaffolding.** "v0.1 - tap to close" is
a placeholder; Palm hid easter eggs (the taxi!) behind About. Cheap joy. Meanwhile **"Add
test events" is in the user-facing Options menu** — gate it behind a build flag or a
five-tap-on-version gesture before anyone else flashes this.

**C6 — Honesty labels.** Two things the device knows but the user doesn't: Memo Pad never
syncs (iCloud has no surface), and To Do syncs to *CalDAV tasks*, not the iPhone Reminders
app (part 11 finding). One quiet line in each app's About/first-run ("Memos stay on this
device") converts future confusion into perceived thoughtfulness. NEXT_STEPS already flags
this; +1, it's cheap and it's the charming *and* honest move.

**C7 — Small authenticity touches worth batching:** ✓ glyph instead of `[x]` in the To Do
column (the Palm font has one); untimed events pinned above timed ones in Day view (Palm
convention — verify current string-sort does this); the classic inverted title bar (Palm's
title was white-on-black; yours is black-on-white with a rule — check against PumpkinOS
screenshots and pick deliberately); battery % + the little battery glyph in the title bar
once U8 lands.

## 4. Intuitiveness (first run and daily loop)

**I1 — The setup path is the product's front door, and it's currently a locked service
entrance.** To go from unboxing to syncing, a user must either edit `secrets.h` and reflash,
or hand-write `config.ini` onto the SD, or Graffiti-enter a Wi-Fi password and a 19-char
app-specific password with untuned templates and no ink. Recommendations, in order of value:

1. **Bless the SD-card path as the official onboarding** (it already works!): polish
   `config.ini.example` into a friendly, heavily-commented template, document "copy this to
   the card, edit two lines, insert, boot" as step 1 in the README, and make the device show
   a *specific* first-boot hint ("No config found — see card setup, or tap Prefs") instead of
   demo data with no explanation.
2. **On-screen keyboard fallback for text fields** — the roadmap (U5) promised "keyboard
   first (works before Graffiti)… so text entry never depends solely on it," but `ui.c` is
   explicitly Graffiti-only. For *password fields specifically*, an lv_buttonmatrix keyboard
   (the calculator pattern — a single object, proven safe in your pool) is the difference
   between possible and painful. Graffiti stays the signature input for everything else.
3. **Validate before trusting**: after Prefs edits, offer "Test connection" (the existing
   login-probe path) so users learn *which* credential is wrong while they're still holding
   the stylus, not after a failed sync.

**I2 — Demo seed data vs. real data.** Seeding demo PDBs on first boot gives the apps life
(good instinct), but a first sync then pushes *Johnny Appleseed and 14 fake meetings into
the user's real iCloud* unless they know to delete them. NEXT_STEPS item 14 is the symptom.
**Recommendation:** mark seeds (a demo category or a `seeded` flag file) and offer
"Remove demo data" before the first HotSync — or simply don't push records that carry the
demo marker.

**I3 — Navigation is close to Palm-right.** Home/Menu/Find/Calc on the silkscreen, list →
detail → edit, category trigger — all good. Two gaps: there's no **Week view** or **Go to
Date** (Palm's fastest navigation); and hardware buttons (the classic date/address/todo/memo
launchers) don't exist on a CYD, so consider long-press silkscreen gestures as the spiritual
equivalent later. Neither is urgent.

**I4 — Feedback loops.** Every action that takes >300 ms should visibly acknowledge: the
sync does (status line), Discover does (spinner text), but Save-to-config, record save, and
delete are silent-success. A 500 ms toast ("Saved") — or the C3 click — closes the loop.

## 5. Other observations (open-source readiness, code health, security)

**O1 — LICENSE is missing and it isn't optional.** The firmware links PumpkinOS-derived
GPLv3 fonts/icons; UI_ROADMAP itself says "Add a LICENSE (GPLv3) and asset-provenance note
when U3a lands" — U3a landed, the file didn't. Before this repo gets one external star:
add `LICENSE` (GPLv3 for the firmware), a `NOTICE`/provenance section naming PumpkinOS and
which assets came from it, and decide + document the host `bridge/` licensing (currently
described as "clean-room and separable" — if you want it more permissive, say MIT/Apache-2.0
explicitly, per-directory). Also: **"Palm" is somebody's trademark** — the project name is
fine as homage, but a README line clarifying non-affiliation is standard practice.

**O2 — No CI.** `make test` + `ftest` (ASan/UBSan) run with zero infrastructure, and
`tests/gate.sh` shows Radicale can be brought up reliably; an ESP-IDF compile check runs in
Docker (`espressif/idf:v5.5`). A single GitHub Actions workflow covering (a) host gates,
(b) fuzz gate, (c) `idf.py build` would catch the "host-green but firmware won't compile"
class before flash day — which your log shows has happened.

**O3 — README serves the developer, not the newcomer.** It's an excellent engineering
record, but the consumer-product framing needs: what this is (one paragraph + a photo),
what you need (CYD ~$15, SD card, iCloud app-password), 5-step setup, then the deep-dive.
Move the war stories to docs/. Keep them — they're gold for contributors — just not on the
front page.

**O4 — Repo hygiene.** `addr_prototype.c` / `pdb_prototype.c` at the root are superseded
scaffolding — delete or move to `attic/`. `app_main.c:256-267` (wifi_connect → run_sync) is
dead code — `lvgl_port_run()` never returns; the headless bring-up path deserves deletion or
an `#ifdef HEADLESS_SMOKE`. The debug telemetry gating (NEXT_STEPS item 12: `fprintf(stderr)`
in dav_esp/sync + the five VERBOSE `esp_log_level_set` calls in `hotsync_task`) is still
open — behind `SYNC_DEBUG`, default off for release.

**O5 — Security posture is reasonable for the class; document it.** Plaintext creds on SD is
a defensible tradeoff for a hobbyist PDA (physical access = game over anyway) and the
config example says so — good. Two cheap upgrades: mask the password fields' *values* in
Preferences rows (done — `********`) and in any future log lines; and note in the README
that the iCloud credential is an app-specific password, revocable at appleid.apple.com
without touching the main account — that's genuinely the right architecture and worth
advertising.

**O6 — Detail-view truncation.** `show_detail` renders into a `static char buf[720]`, but a
memo can be 1200 bytes (edit buffer) and notes 500 — long records display truncated with no
ellipsis. Bump to match `mtext` or add a "…" marker.

## 6. What can safely be done in-repo while you're away (no device required)

The test architecture makes an unusual amount of work device-free: everything in `bridge/`
is host-gated, firmware changes can be *compile-verified* with `idf.py build`, and several
past device bugs were reproduced on the host deliberately. Ranked by value:

### Green — fully verifiable offline (build + run host gates)

1. **LICENSE + NOTICE + trademark line + license-of-`bridge/` decision (O1).** Pure repo
   work, highest product impact, zero risk.
2. **CI workflow (O2):** host gates + fuzz + Radicale-backed gates via `tests/gate.sh` +
   `idf.py build` in Docker. Immediately protects all vacation work that follows.
3. **README restructure + real user setup guide (O3, I1.1)** including the polished
   `config.ini` onboarding doc. Words only.
4. **M1: sync BSS → sync-lifetime heap.** Mechanical, and the entire correctness surface is
   covered by `run_gates.sh` (roundtrip/incremental/synctoken/bigsync/idempotent/uidmatch all
   exercise these exact buffers). Compile-check the firmware too. Flash-verify on return.
5. **M4 (`MAX_DISC`), M5 (deferral counts in `SyncStats` + status line), O4 (dead code,
   prototypes, `SYNC_DEBUG` gating), O6 (detail truncation).** Small, host- or
   compile-verifiable, each closes a real gap.
6. **Graffiti recognizer host harness.** `graffiti.c` is pure C with a single `ESP_LOG`
   dependency — factor that behind a macro, build it on the host, and add a gate that feeds
   recorded stroke point-lists (captured earlier from `graf` telemetry, or synthesized) and
   asserts recognition. Then the X two-stroke support, template refinements, and the
   training-game logic can all be *developed and regression-tested offline* — only the
   final threshold numbers need the resistive panel. This converts the project's flakiest
   subsystem into a tested one.
7. **Config/fuzz coverage extensions:** `config.c` fuzz corpus, a vCard-with-PHOTO (>8 KB)
   gate to lock in the M5 behavior, and a malloc-failure-injection wrapper for the sync path
   (assert clean -1, no partial map writes) — the host analog of the fragmentation risk.

### Yellow — build off-device, verify on return (compile-check now, flash later)

8. **An LVGL simulator target for `ui.c`** (LVGL's SDL driver; stub `data.c` over local
   files, stub hotsync). This is the highest-leverage single investment for the UX goals:
   C1 ink trail, C2 HotSync dialog, C4 form-contract rework, C5 About, I2 first-boot hint,
   I4 toasts, and the keyboard fallback (I1.2) all become iterate-in-minutes work on a
   laptop, with `LV_MEM_SIZE=24K` set so pool exhaustion reproduces on the desktop. Even a
   partial port (launcher + one app) pays for itself immediately.
9. **M2 design + implementation** (sync-mode teardown/rebuild), written against the
   simulator + compile-checked; the heap numbers need hardware.
10. **C2/C4/C5/C7 UI work** if the simulator (8) exists; otherwise write + compile-check
    and expect a fix-up pass on hardware.

### Red — do not attempt without the device

- Graffiti *threshold/template tuning* against the real resistive panel (harness in #6
  prepares it; numbers need glass).
- Anything touching touch calibration, display init, light-sleep/PM, brightness, DMA.
- Live-iCloud verifies still pending in NEXT_STEPS (To Do full-reconcile heal, >24-record
  idempotency, relocation trace) and the M3 soak test.
- U8 battery gauge / U9 case.

### Suggested vacation sequencing

Week 1: items 1–5 (a day of product hygiene that permanently de-risks the repo), then start
6. Week 2+: item 8 (simulator), then spend the remaining time doing UX work (C1, C2, C4,
I1.2, I2) *inside* it. Return with: a repo that's legally publishable, CI-protected, a
tested Graffiti core, and a stack of UI improvements awaiting a single flash-and-verify
session.

## Appendix: things checked that are fine

- Stream parsers' window-boundary handling now covered by `streamparse.c` (the vacuous-test
  fix in part 11 was the right catch).
- `s_prog` cross-task int is a single aligned write on ESP32 — safe as used.
- Conflict policies, UID-keyed identity, delete-suppression-on-unresolved, and the
  empty-result overwrite guard are all the right calls and all gated.
- The `lv_table`-not-`lv_list` and `VALUE_CHANGED`-not-`CLICKED` findings are correct LVGL 9.2
  behavior and well documented in-line.
- Partial draw buffer (40 rows, DMA-capable) and the mono theme init are textbook for this
  panel.
